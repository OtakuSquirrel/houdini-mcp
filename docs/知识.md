# Houdini MCP 技术架构知识库

## 1. 整体架构

项目由三个独立层组成，各层使用不同的通信协议：

```
MCP Client (Agent)  ──[MCP传输层]──►  MCP Server (houdini-mcp)  ──[RPyC/TCP]──►  Houdini
                                              │
                                              ▼
                                     WebUI (FastAPI/uvicorn, 可选)
```

## 2. 技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| MCP Server 框架 | **FastMCP v2.0+** | Python MCP SDK，处理 MCP 协议通信 |
| MCP 传输层 | **stdio**（当前） | 客户端通过子进程 stdin/stdout 通信 |
| Houdini 通信 | **RPyC v4.x (classic mode)** | TCP socket 远程过程调用，获取远程 `hou` 模块代理 |
| RPyC 默认端口 | **18811**（范围 18811-18899） | 支持端口自动发现和动态分配 |
| WebUI | **FastAPI + uvicorn** | 可选管理界面，端口 8765 |
| 构建系统 | **hatchling** | pyproject.toml 配置 |

### 关键依赖

- `fastmcp>=2.0` — MCP 协议实现
- `rpyc>=4.1,<5` — Houdini 远程控制
- `Pillow>=10.0` — 图像处理
- `scikit-image>=0.22` — 截图比较 (SSIM)
- `pywin32>=306` — Windows 窗口截图
- `fastapi>=0.110` + `uvicorn>=0.27` — WebUI（可选依赖）

## 3. MCP 传输层选项

FastMCP 支持 4 种传输方式，与 WebSocket 平行：

| 传输方式 | 连接模型 | 当前使用 | 适用场景 |
|---------|---------|---------|---------|
| **stdio** | 子进程 stdin/stdout | **是** | 本机、单客户端、零配置 |
| **SSE** | HTTP 长连接 + Server-Sent Events | 否 | 需要持久连接的远程场景 |
| **streamable-http** | HTTP POST 流式响应 | 否 | 远程、多客户端、推荐的新方案 |
| **http** | 标准 HTTP POST | 否 | 无状态请求 |

> 注意：MCP 协议本身没有原生 WebSocket 传输，但 SSE 和 streamable-http 在功能上覆盖了 WebSocket 的大部分场景。

### stdio 的优缺点

**优点：**
- 零配置，不需要管端口/地址/认证
- 天然隔离，每个客户端一个进程
- 安全，没有网络暴露面

**局限：**
- 1:1 绑定，一个客户端启动一个子进程，不能共享
- 仅限本机，不支持远程连接
- 生命周期耦合，客户端退出则 MCP Server 也退出

### 切换传输方式

切换到网络传输只需改一行代码：

```python
# __main__.py 中 mcp.run() 改为：
mcp.run(transport="streamable-http", host="0.0.0.0", port=8080)
```

## 4. RPyC 连接层

### 工作原理

- Houdini 端：通过 `houdini_plugin/houdini_mcp_startup.py` 在 456.py 中注入钩子，启动 `hrpyc.start_server()` 监听 TCP 端口
- MCP Server 端：通过 `rpyc.classic.connect(host, port)` 连接，获取远程 `hou` 模块完整访问权限
- 连接管理：`ConnectionManager` 支持多 Houdini 实例管理（多端口、多连接、活跃连接切换）

### 端口发现机制（优先级）

1. CLI `--port` 参数（显式指定）
2. `HOUDINI_MCP_PORT` 环境变量
3. 自动扫描端口范围（18811-18899）发现未被占用的 RPyC 监听
4. 回退到默认端口 18811

### 安全警告

RPyC classic mode 是**零认证全权限**模式，连接方拥有远程 Python 完整执行权限（等价于 SSH root）。在 localhost 使用没有问题，但绝不能直接暴露到公网。

## 5. Docker 化与远程部署分析

### 部署拓扑

```
┌─ Docker ─────────────────┐     ┌─ 远端服务器 ──────────────┐
│                          │     │                           │
│  Agent (Claude Code)     │     │  Houdini + RPyC listener  │
│    │                     │     │    ▲                      │
│    │ streamable-http     │     │    │ RPyC/TCP             │
│    ▼                     │     │    │                      │
│  MCP Server ─────────────┼─────┼────┘                      │
│   (houdini-mcp)          │     │                           │
└──────────────────────────┘     └───────────────────────────┘
```

### 远程部署需要的改动

| 层 | 改动内容 | 涉及文件 | 难度 |
|----|---------|---------|------|
| MCP 传输 | `mcp.run(transport="streamable-http")` | `__main__.py` | 一行代码 |
| RPyC 连接 | `host` 参数从 `localhost` 改为可配置的远端地址 | `connection.py` | 小改 |
| 生命周期管理 | `start_houdini`/`stop_houdini` 拆分本地模式和远程模式 | `lifecycle.py` | 中等 |
| 屏幕截图 | pywin32 截本地窗口远程失效，需改用 Houdini 内置渲染或远端截图服务 | `tools/screen.py` | 中等 |
| 安全 | RPyC classic mode 零认证，暴露到网络极其危险 | 全局 | 重要 |

### 协议层能覆盖什么

- **streamable-http** 完全覆盖 MCP 传输层的远程需求
- **RPyC/TCP** 天然支持远程连接（改 host 参数即可）
- **协议层不是瓶颈**——真正需要重构的是 lifecycle 管理（本地进程操作）和安全模型

### 安全层必须解决的问题

远程部署时必须加以下至少一项：
- RPyC 切换到 `rpyc.Service` 模式（白名单 API，放弃 classic mode）
- SSH 隧道 / VPN（网络层隔离）
- 在 RPyC 前面加认证代理 + TLS

## 7. 踩坑记录：Agent 启动 Houdini 慢的问题

### 背景
Agent 通过 MCP 启动 Houdini 需要 50 秒甚至 120 秒超时，而手动启动只需 11 秒。

### 根因 1：456.py 包含旧版完整启动脚本（端口不匹配）
- `456.py` 被错误地写入了 261 行的旧版启动脚本（硬编码 `PORT = 18811`），末尾还追加了 hook 行
- Agent 动态分配端口（如 18815），旧代码却在 18811 启动 RPyC → Agent 轮询错误端口直到 120s 超时
- **修复**：456.py 只保留一行 hook：`exec(open(hou.homeHoudiniDirectory() + "/scripts/houdini_mcp_startup.py").read())  # [HoudiniMCP]`

### 根因 2：subprocess.Popen + DETACHED_PROCESS + DEVNULL 拖慢 Houdini 初始化
- `subprocess.Popen` 使用 `creationflags=DETACHED_PROCESS | CREATE_NEW_PROCESS_GROUP` + `stdout/stderr=DEVNULL` 启动 `houdinifx.exe`
- 实测：同样的 exe，命令行直接启动 11 秒，Popen 方式 30+ 秒
- 去掉 `DETACHED_PROCESS` → 进程被父进程句柄阻塞（按 ESC 取消后才启动）
- **修复**：改用 `start ""` shell 命令启动 GUI，和双击行为一致
- **注意**：`start` 立即退出，`proc` 是 `cmd.exe` 而非 Houdini，轮询循环需跳过 `proc.poll()` 退出检查

### 根因 3：轮询间隔过大
- 原来每轮 `sleep(2)` + socket timeout `1s` + 宽限期 `1s` ≈ 4 秒/轮
- **修复**：`sleep(0.5)` + socket timeout `0.3s` + 无宽限期 ≈ 0.8 秒/轮

### 最终结果
修复前 120s 超时失败 → 50s → **11s**（与手动启动一致）
